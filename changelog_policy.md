# Changelog Policy

> Единый источник правил создания и обновления `CHANGELOG.md`.

## Scope

- Файл: `CHANGELOG.md`
- Формат: **Keep a Changelog** (без секции `Unreleased`)

## Entry order

- История изменений ведётся **сверху вниз**.
- Самая новая версия добавляется **в начало списка версий** (сразу после заголовка `# Changelog`).

## Version header format

Каждая запись версии должна иметь заголовок в формате:

- `## [X.Y.Z.T] - YYYY-MM-DD HH:mm`

Где:
- `X.Y.Z.T` — версия по правилам `version_policy.md`;
- `YYYY-MM-DD HH:mm` — дата и время создания записи (локальное время репозитория или время, согласованное в команде).

## Keep a Changelog sections

Разрешённые секции внутри версии:
- `### Added`
- `### Changed`
- `### Fixed`
- `### Removed`
- `### Security`

Секции добавляются только при наличии реальных изменений соответствующего типа.

## Mandatory summary rule

В каждой записи версии **обязательно первым пунктом** добавляется общий смысл изменений:

- секция: `### Summary`
- 1–3 пункта, описывающих суть релиза на уровне продукта/функциональности.

Пример:
- `### Summary`
- `- Улучшена стабильность упаковки NuGet и синхронизация метаданных.`

## Noise filtering

В `CHANGELOG.md` запрещено добавлять банальные/технически шумовые записи, например:
- «обновили changelog»;
- «исправили форматирование» (если это не влияет на поведение/контракт);
- «переставили строки/комментарии без функционального эффекта».

Записываются только значимые изменения для пользователей/разработчиков библиотеки.

## Source of truth

- Правила содержания и структуры: `changelog_policy.md`
- Правила версии: `version_policy.md`
