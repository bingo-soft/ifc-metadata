# Skill: Finalize Release and Merge

## Purpose

Выполнить финализацию изменений по явной команде пользователя:

- `завершить`
- `завершить и мерж`

В обычных задачах (без этих команд) skill не применяется.

## Command triggers

- `завершить`
  - режим: `finalize`
- `завершить и мерж`
  - режим: `finalize+merge`

## Default behavior (without trigger)

Если пользователь не дал одну из команд-триггеров:

- не менять версию;
- не обновлять `CHANGELOG.md`;
- не делать commit;
- не создавать tag.

## Versioning source of truth

Правила версионирования и расчёта следующей версии берутся из:
- `version_policy.md`

## Flow: "завершить"

1. Определить список фактических изменений в рабочем дереве.
2. Обновить версию проекта по правилам `version_policy.md`.
3. Сформировать запись в `CHANGELOG.md`:
   - дата;
   - added/changed/fixed (или добавлено/изменено/исправлено).
4. Проверить необходимость обновления `README.md`:
   - если поведение/контракты/публичные команды изменились — обновить;
   - иначе оставить без изменений.
5. Подготовить commit по policy:
   - формат сообщения: `<type>: <short summary>`.
6. Создать tag по версии (формат тега — по правилам `version_policy.md` или принятый в репозитории формат).
   - В имени тега не использовать префикс `v` (пример: `2.15.10.47`, не `v2.15.10.47`).

## Flow: "завершить и мерж"

1. Выполнить весь flow команды `завершить`.
2. Определить текущую git-ветку.
3. Если текущая ветка `develop`:
   - merge пропустить.
4. Если текущая ветка не `develop`:
   - выполнить merge текущей ветки в `develop`.

## Validation before commit

- Не запускать сборку/тесты (если есть).
- Не коммитить, если сборка падает.

## Commit policy

- Формат: `<type>: <short summary>`
- Пример: `feat: add memory/performance profiling for large IFC`
