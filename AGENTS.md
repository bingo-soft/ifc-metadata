# Repository agent instructions

## Режим работы по умолчанию

При обычных изменениях через Codex (без специальных команд):

1. Не обновлять версию проекта.
2. Не изменять `CHANGELOG.md`.
3. Не делать commit.
4. Делать только запрошенные пользователем рабочие изменения.

## Команда: "завершить"

По явной команде пользователя `завершить` агент обязан:

1. Обновить версию проекта по правилу из `version_policy.md`.

2. Собрать информацию для `CHANGELOG.md` и добавить запись:

   - дата;
   - что добавлено/изменено/исправлено.
3. Если нужно по смыслу изменений — обновить `README.md`.
4. Подготовить commit.
5. Создать git tag с версией.

## Команда: "завершить и мерж"

По явной команде пользователя `завершить и мерж` агент обязан:

1. Выполнить все шаги команды `завершить`.
2. Определить текущую ветку.
3. Если текущая ветка не `develop` — выполнить merge в `develop`.
4. Если текущая ветка `develop` — merge не требуется.

## Versioning source of truth

Правила версионирования определены в:
- `version_policy.md`

## Связанные skills

Детальные пошаговые процедуры описаны в файлах:
- `.codex/skills/finalize-and-merge.md`
- `.codex/skills/changelog-create-and-update.md`

## Commit policy
- Формат: `<type>: <short summary>`
- Пример: `feat: add memory/performance profiling for large IFC`
- В commit включать все изменения, которые не попадают под `gitignore`.


## Validation before commit
- Не запускать сборку/тесты (если есть).
- Не коммитить, если сборка падает.

## Code
1 - Где необходимо используй switch expression, или switch или оставляй multiple if
2 - Где есть extension method, используй extension block как в .NET 10
3 - Подробные комментарии писать только после пункта 2

## Примечание

Команды `завершить` и `завершить и мерж` выполняются только по явному запросу пользователя в текущем диалоге.
